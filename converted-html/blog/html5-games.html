HTML5 Games:
Or, how I learned to stop worrying and love the browser


I ran a little training session for some of the other Producers and Produciton assisstants at Aardman about what I knew about making HTML5 games...this is it!

We’re going to learn about making 2D games with Phaser, but there are plenty of other ways to make online games!

First of all…

  • They are not really ‘HTML5 games’! This is just a catch all term invented by the ‘media’. HTML5 is the most recent HTML spec, with new features and ways of doing things.

  • They are actually comprised of lots of browser features like WebGL, JavaScript, GLSL, Web Audio, Canvas, DOM, CSS3 etc…

When should I make an HTML5 games?

  • When you want a game that can run in the browser on desktop, tablet, and mobile.

  • You can also make an app by wrapping the code up in something like CocoonJS

  • …but if an app is your primary platform then you should probably be thinking about Unity, with an option of the WebGL export for a browser version.

What if I want to support IE8?

  • Then you will need to do everything in the DOM (see below) and not use canvas; this means NO PHASER!

  • But don’t do this…IE8 is OLD. Some schools might be stuck on this because of archaic IT policy and machines, but it is terrible.

So what is an ‘html5’ game made of?

Phaser

  • Is a game framework developed by our ex-tech direction, Rich Davey. It’s very good.

  • It can deal with lots of the stuff you will need to do out of the box, and is completely free!

HTML, CSS, JS: THE BUILD

  • The live build of your game will be comprised to three elements:

    1. HTML: this is the actual nodes on the page, the building blocks on any webpage. In a Phaser game this will be a <canvas> element with a width and a height, which is a lump of plain graphics data of that size. You might add other HTML things over the top if you like: text forms, alerts, etc. This what people are referring to when they talk about the DOM (Document Object Model). In BBC games, things like the ‘Rotate your Device’ message have to be HTML. You can create whole games in the DOM, using just HTML and CSS, but that is generally if you need to support something weird.

    2. CSS: this is used for styling HTML elements, which otherwise just be rendered as the browser draws them. This would most normally be used for styling the embed of your game, or any other HTML elements you have included.

    3. JS: and this does all the hardwork! Javascript creates, renders and manipulates everything inside your HTML <canvas> element.

    4. JSON: is basically a spreadsheet. It should store all your text content which is rendered in the game, and also config settings, so things can be changed easily by your or the client without having to re-build the thing. This is especially good with the BBC, as you can tweak text and configuration without having to go through tech review again!

What’s all this WebGL nonsense then?

  • WebGL is a rendering engine that allows the browser to have GPU accelerated graphics in a web-game.

  • The GPU (Graphics Processing Unit) is part of a computer or device specifically designed for drawing graphics quickly and efficiently, and thus removes the burden from the CPU
    • (central processing unit).

  • WebGL can also do lots of cool stuff that a normal canvas element cannot:

    1. GLSL, which is a shader language allowing you to do cool visual effects over all or part of your canvas. See some examples here: http://www.goodboydigital.com/pixijs/examples/15/indexAll.html

    2. 3D! Using something like three.js: http://threejs.org/

    3. Dynamic lighting! Thought we’ve never tried this yet, http://www.goodboydigital.com/pixijs/pixilights/

    4. Boned animation! Little bits of asset, which are then manipulated on the fly using maths.

    5. Probably loads more!

    So how do I use it? If you are using Phaser, you are! Phaser uses the pixi.js renderer, which uses WebGL, and falls back to canvas if it is not available.

    Typescript, grunt, assets, etc: THE SOURCE

  • Typescript: compiles to javascript, but is statically typed. (I think) this means it checks at compile time whether you are passing the correct type of thing into your variable (number, string, etc), and tells you if there is a problem. There are probably other things, but this is what I know about it.

  • Grunt: is a taskrunner, and what we use to build the project. It takes all the source files and turns them into the build, and you can configure it do any part of the process you like e.g.

    1. Compile TS to JS

    2. RequireJS (which loads in modules, librarys etc into one file)

    3. Remove unneeded files

    4. Minification (removing all the whitespace from your code to make it smaller)

    5. Optimise images

    6. FTP deploy to the server

    Images

  • Images: graphics will be sprite based, usually PNGs

  • Animations will be sequences of PNGs that play one after another

  • All your images should be compiled into (preferably) one spritesheet – a big sheet of all the images, and a reference in the game. Use a program like TexturePacker to squeeze them in as tightly as possible.

    1. Why one spritesheet? Unloading and loading sheets into memory is processor intensive for a device. If it can just load one into memory and that’s it, amazing!

    2. Spritehsheets should be Power of 2 e.g. 512 x 512, 1024 x 1024, 2048 x 2048, etc. This is the sizes that GPUs use, and should make your game more performant (though if not using WebGL it doesn’t matter).

    Sound

  • Is a lot better than it used to be!

  • Web Audio supports polyphonic sound, effects like reverb and more

  • Internet Explorer and older Android browsers still an issue - Phaser sound falls back to the older Audio API, so it can only play one sound at a time

  • Should effects should be concatenated into sound sprites, which is one long track with all the sounds in, with a time reference for each one.

  • You need MP3 and OGG versions of your sounds.

  • Music should be as individual looping sounds, and kept as short as possible.

  • Use Soundsprite to create them: https://www.npmjs.com/package/soundsprite

    • Other things!

    Libraries:

  • There are plenty of good open source javascript libraries for doing lots of cool things. Some we have used on DMK include:

    1. Hammer: for touch gestures

    2. Underscore: extends javascript with lots of new functionality you don’t need to write

    3. jQuery: same as above

    4. JSONpack: for compressing JSON files so they could be emailed (long story)

    IDEs for Javascript/Typescript:

  • If they use something like IntelliJ or Visual Studio Code will make life a lot easier for your developers: error highlighting, lookups, hints for Typescript.

    • Things to be aware of…

  • Performance is always an issue on devices. Save lots of time for optimising your game, and make sure you test it on devices throughout the build process, not just at the end!

  • iOS for some reason needs a touch/click event before it will play any sound. Make sure you have one of these before anything important happens in your game!

  • All browsers will perform differently, so again, leave lots of time for testing. REMEMBER that Twitter and Facebook both have their own internal browsers now. 50% of the mobile traffic to Eastenders Storyteller is the Facebook internal browser! FFS!

  • Accessibility for the blind (e.g. tab controls and screen readers) has been bad for a while in Phaser and canvas games in general, as the tools rely on the DOM to be read. The way to get around this is to make all the UI as DOM elements and overlay the phaser canvas with them. (See Something Special and Deadly Defenders.)

  • File size, mobile games may be loaded over mobile networks and we should take care to not over use end users’ data plans. Libraries like require js can help by minifying and combining all the javascript, but content is the real killer.

  • Scope, The DMK projects are the exception as opposed to the rule. It is best to have a good mechanic that is well polished than lots of content, which will damage performance, and increase filesize. KISS!
